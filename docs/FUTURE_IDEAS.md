# Future Ideas

Ідеї для майбутнього обговорення. Не архітектурні рішення, не ADR - рівень Discussion, ще не Architecture Decision.

## Age-Typicality Score vs Abnormality Score (два різні типи "аномальності")

**Проблема.** abnormality_score() відповідає на питання "наскільки число відхиляється від норми для цієї людини" (вік/стать враховані через Reference Range). Існує другий, окремий тип аномальності: "наскільки типовим чи рідкісним є сам факт наявності стану/діагнозу для людини такого віку" (приклад: артроз у 18 років - аномальний не через число з аналізу, а через сам факт діагнозу в невідповідному віці).

**Чому це не Reference Range і не Laboratory.** Reference Range працює з числовими показниками одного аналізу. Ця ідея працює з фактом наявності стану - територія майбутнього модуля Diseases, ще не спроєктованого.

**Можлива майбутня назва концепції:** "Age-Typicality Score" чи "Epidemiological Rarity" - оцінка рідкісності діагнозу для вікової групи пацієнта, на основі епідеміологічних даних (Disease Criteria + вікові розподіли захворюваності).

**Статус:** зафіксовано для майбутнього обговорення при проєктуванні модуля Diseases. Перед реалізацією - підготувати план і провести Discussion з ChatGPT (Devil Advocate) до Architecture Decision.

## Clinical Context Override (медична історія змінює інтерпретацію норми)

**Проблема.** Reference Range враховує лише демографію (стать, вік). Але є ситуації, коли медична історія пацієнта (видалений орган, встановлений діагноз, прийом препаратів) робить "аномальне" значення очікуваним і нормальним для цієї конкретної людини. Приклад: ТТГ після видалення щитоподібної залози.

**Чому це не Reference Range.** Це вимагає знання медичної історії пацієнта (операції, діагнози, препарати) - інформації, якої Laboratory-модуль немає і не повинен мати напряму.

**Залежність.** Реалізація можлива лише після появи модулів Diseases/Medical History та Medications, і механізму зв'язку між конкретним лабораторним показником і конкретним медичним фактом.

**Статус:** зафіксовано для майбутнього обговорення. Не задача поточного етапу.

## Multi-Lab Data Normalization (test_code стандартизація між лабораторіями)

**Проблема.** Reference Range Resolver v1 підтверджений на реальних даних однієї лабораторії (Діла). Дані від інших лабораторій можуть використовувати інші коди тестів для того самого показника, інші назви методів, інші референсні джерела.

**Що вже готово.** Архітектура test_code+unit+sex+age+laboratory_name+method вже дозволяє співіснування кількох джерел одного показника без конфлікту (підтверджено: generic TSH і high-sensitive TSH мирно співіснують у довіднику).

**Що не вирішено.** Чи потрібен окремий шар "аліасів" кодів тестів між лабораторіями (наприклад, прив'язка до майбутнього LOINC-стандарту з Constitution "Довгострокове бачення")? Це вимагає реального другого прикладу лабораторії з іншим кодуванням, перш ніж проєктувати рішення.

**Уточнення (реагенти як контекст пояснення, не поле схеми).** Навіть без структурованого поля reagent/analyzer, шар інтерпретації (Domain чи майбутній AI-Consilium) повинен розуміти, що той самий показник з різних лабораторій може відрізнятись через різні реагенти/методики - і чесно повідомляти про це користувачу при порівнянні результатів (наприклад, у trend()), а не мовчки показувати "погіршення" чи "покращення" без цього застереження. Це знання для інтерпретації, не дані для зберігання.

**Статус:** зафіксовано. Потребує окремого Discussion + Devil Review (ChatGPT) з реальним прикладом другої лабораторії, перш ніж будь-яка реалізація. Не задача поточного етапу.

## Three Levels of Thinking (candidate architecture principle, not yet formalized)

**Джерело:** Devil Review round 5 (ChatGPT), після Sprint 3 (Reference Range + Imaging).

**Спостереження.** FHOS складається не лише з модулів, а з трьох рівнів мислення:

- **Level 1 - Data.** Laboratory, Imaging, Profile, майбутні Medications/Procedures. Відповідають лише за факти ("LDL = 5.3 mmol/L", "MRI: L4-L5 disc protrusion"). Не мають права робити клінічні висновки ("це небезпечно", "це пояснює симптоми").
- **Level 2 - Knowledge.** Reference Ranges, майбутні Drug Interactions, Disease Criteria, Clinical Guidelines, LOINC, ICD, SNOMED. Описують медичний світ узагальнено ("У медицині прийнято вважати..."), не виносять суджень про конкретного пацієнта.
- **Level 3 - Reasoning.** Єдиний рівень, де дозволено поєднувати дані з кількох Data/Knowledge джерел і робити висновки ("Разом ці дані можуть свідчити..."). Тут у майбутньому житимуть AI Consilium, Devil, Composite Clinical Risk Engine.

**Ключове правило (запропоноване, не Approved):** "No medical module may perform reasoning that requires information owned by another medical module." Якщо логіка (Python, SQL, if/else, prompt - будь-що) потребує даних більш ніж одного медичного модуля - це вже Clinical Reasoning, і має жити тільки на окремому рівні, не всередині модуля.

**Чому це важливо.** Без цієї межі технічний борг накопичується непомітно: спочатку один if, що поєднує Laboratory з Imaging, потім другий - і за рік кожен модуль "знає" суміжні медичні спеціальності, стаючи взаємозалежним монолітом.

**Перевірка проти існуючого коду (Sprint 3, момент фіксації ідеї):** жодного порушення не знайдено. interpret()/abnormality_score()/trend() оперують даними виключно свого модуля (Laboratory). ReferenceRangeResolver поєднує Data (Laboratory) зі своїм Knowledge (Reference Range) - це не порушення, бо правило забороняє Data+Data між різними медичними модулями, не Data+Knowledge для одного факту.

**Статус:** зафіксовано як кандидат на фундаментальний принцип. НЕ переноситься в Constitution зараз. Рекомендація (ChatGPT, підтримана): почекати підтвердження на 2-3 незалежних майбутніх модулях (Medications, Diseases, AI Consilium) - "спочатку практика, потім правило", той самий підхід, що вже застосований до Knowledge Layer.
