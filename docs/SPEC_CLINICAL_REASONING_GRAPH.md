# Clinical Reasoning Graph — Специфікація (S07E09, Draft)

Статус: Architect Session у процесі. Це НЕ Implementation Mode.
Жодного коду не пишемо, поки не вирішено Data Source (розділ 9).

## 1. Головний принцип

Медична мережа зв'язків (Clinical Reasoning Graph) НЕ будується з
аналізів пацієнта і не росте безконтрольно. Вона існує окремо, як
готова, попередньо визначена структура. Дані пацієнта -- це
Patient Overlay: шар, що підсвічує вузли мережі та їхнє найближче
оточення, не будує мережу заново щоразу.

Ключова відмінність від Knowledge Base (Contraindications, Drug
Interactions): Knowledge Base відповідає "чи є ця пара небезпечною"
(готова відповідь так/ні). Clinical Reasoning Graph відповідає
"що взагалі могло б це пояснювати" (список гіпотез для перевірки).
Це різні задачі, різні структури даних, не варіації одна одної.

## 2. Типи вузлів (Node Types)

- Organ / AnatomicalStructure -- орган чи анатомічна структура
- Function -- фізіологічна функція (обмін речовин, кровообіг)
- Biomarker -- вимірюваний показник (ТТГ, холестерин, сечова кислота)
- Symptom -- скарга пацієнта (біль, втома)
- Disease -- діагноз (посилання на MONDO ID, той самий підхід,
  що вже є в Diseases/Contraindications)
- Drug -- препарат (посилання на CHEBI ID, той самий підхід,
  що вже є в Medications/Drug Interactions)
- LifestyleFactor -- спосіб життя (малорухливість, харчування, сон)

## 3. Типи зв'язків (Edge Types)

Зв'язок описує НЕ "що лікує що", а "що МОЖЕ пояснювати що" (ChatGPT,
S07E09). Кожне ребро графа має:

- relationship_kind: enum
  - CAN_EXPLAIN (симптом -> можлива причина)
  - AFFECTS (показник A впливає на показник B)
  - ASSOCIATED_WITH (кореляція без встановленого напрямку причини)
  - CONTRAINDICATES (уже покрито окремим модулем Contraindications,
    сюди НЕ дублюється)
- direction: чи є напрямок причини встановленим, чи це двостороння
  асоціація (AFFECTS завжди напрямлений, ASSOCIATED_WITH -- ні)
- evidence_level: посилання на розділ 4
- source_citation: DOI/URL конкретного дослідження

## 4. Рівні довіри (Evidence Levels)

Чотирирівнева шкала (ChatGPT, S07E09), приймається як canonical:

| Рівень | Джерело | Позначення |
|---|---|---|
| 1 | Міжнародні клінічні рекомендації (EAU, ESC, ADA, ATA) | ★★★★★ |
| 2 | Систематичні огляди, мета-аналізи, Cochrane | ★★★★☆ |
| 3 | Окремі дослідження (RCT) | ★★★☆☆ |
| 4 | Клінічний досвід, кейс-репорти, конференції | ★★☆☆☆ |

Кожне ребро графа зобов'язане мати evidence_level. Немає "ребер без
джерела" -- якщо джерела нема, зв'язок не є частиною графа, а є
Hypothesis (розділ 5).

## 5. Життєвий цикл гіпотези (Hypothesis Lifecycle)

```
Hypothesis -> Verification -> Evidence -> Accepted Knowledge -> Rejected
```

Нові зв'язки (з конференцій, кейс-репортів, спостережень) НЕ
потрапляють у граф автоматично. Вони існують як Hypothesis із явним
evidence_level=4, доки не пройдуть незалежну перевірку. AI (ні
Claude, ні майбутній Consilium-модуль) не має права самостійно
підвищувати Hypothesis до Accepted Knowledge -- це рішення людини
(лікаря/експерта), той самий принцип, що Governance Lifecycle
(ADR-0008) для архітектурних рішень проекту, тепер застосований до
медичних гіпотез.

## 6. Patient Overlay -- алгоритм (концептуально, не код)

1. Взяти всі активні (is_active()) записи пацієнта: Medications,
   Diseases, LaboratoryResult з відхиленням від норми.
2. Для кожного запису знайти відповідний вузол у графі (той самий
   принцип нормалізації, що вже є: CHEBI для препаратів, MONDO для
   хвороб -- SUBSTANCE_TO_CHEBI/DISEASE_TO_MONDO вже існують).
3. "Підсвітити" знайдений вузол і ВСІХ його прямих сусідів
   (за CAN_EXPLAIN/AFFECTS/ASSOCIATED_WITH ребрами).
4. Результат -- підмножина графа, релевантна для цього пацієнта
   в цей момент, НЕ весь граф.

## 7. Hypothesis Expansion -- алгоритм (концептуально, не код)

Для кожного підсвіченого Symptom/Biomarker-вузла:

1. Знайти ВСІ вихідні CAN_EXPLAIN ребра (не тільки одне очевидне).
2. Не зупинятись на першій правдоподібній причині (anchoring bias,
   реальний приклад -- hip-spine syndrome, S07E09).
3. Ранжувати кандидатів за evidence_level, потім за тим, чи вже є
   в пацієнта підтверджуючі/спростовуючі дані (Patient Overlay).
4. Розділити на: (а) підтверджено даними пацієнта, (б) можлива
   причина, дані відсутні -- сформулювати явний запит на дані.

## 8. Devil Review -- чек-лист (ChatGPT, S07E09, canonical)

Три обов'язкові питання при кожному циклі аналізу:

1. "Які пов'язані напрямки консиліум НЕ перевірив?"
2. "Які альтернативні причини можуть пояснити картину?"
3. "Які органи або системи залишились поза увагою?"

Метрика успіху НЕ "краса відповіді", а повнота охоплення графа
(скільки релевантних CAN_EXPLAIN-гілок розглянуто з тих, що існують).

## 9. Кандидати джерел даних (перевірено реальним пошуком, S07E09)

Рівні 1-3 (готові структуровані онтології):
- MONDO (вже використовується в Contraindications) -- хвороби
- HPO (Human Phenotype Ontology) -- симптоми/фенотипи, НЕ хвороби
  (раніше в MeDIC-даних відкидали HP-рядки як "не хвороба" --
  насправді це окрема потрібна структура, не сміття)
- DO-KB (Disease Ontology Knowledgebase) -- 20-річний проект, прямо
  описує "від ознак хвороби до диференціальної діагностики"
- Disease-Symptom structured datasets (напр. arXiv 2506.13610) --
  табличні дані хвороба<->симптом із рецензованих джерел

Рівень 4 (кейс-репорти):
- MultiCaRe Dataset (DOI 10.1016/j.dib.2023.110008) -- 75382
  відкритих кейс-репорти з PMC Open Access, деідентифіковані
- 94% журналів з кейс-репортами -- відкритого доступу (PMC, BMC)

Жодне з цих джерел ще НЕ перевірено на точний формат/ліцензію так
детально, як MeDIC/Phansalkar/MONDO раніше в проекті -- це перший
крок, коли розпочнеться реальна робота над S07E09.

## 10. Явно поза Scope (не вирішено, не забуто)

- Психіка/менталітет -- окрема, складніша територія доказовості,
  додається пізніше, тим самим методом (Hypothesis Lifecycle), не
  зараз (власник продукту, S07E09).
- Вибір технології зберігання графа (графова БД типу Neo4j, чи
  property graph у PostgreSQL, чи щось інше) -- НЕ вирішено, окреме
  технічне рішення, коли дійде до Implementation Mode.
- Механізм оновлення (нові конференції/дослідження з часом) --
  сформульована потреба (власник продукту), конкретний механізм
  не спроєктований.
- Interaction Evidence View (Drug Interactions) і цей граф --
  окремі механізми, поки що не інтегровані між собою.
