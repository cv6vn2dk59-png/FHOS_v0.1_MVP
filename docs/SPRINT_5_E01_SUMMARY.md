# FHOS S05E01 — Drug Interactions — Architect Session Summary

Дата: 2026-07-06
Статус: Architect Mode завершено. Код ще не писали.

## Рішення

1. **Scope v1**: тільки препарат ↔ препарат. Не робимо зараз:
   препарат-їжа, препарат-алкоголь, препарат-хвороба, препарат-аналіз,
   препарат-генетика — кожен має власне джерело даних.

2. **Ідентифікатор речовини**: назва діючої речовини (INN), не ATC-код
   і не назва бренду. ATC лишається додатковою інформацією для
   класифікації, не головним ключем — бо ATC не розрізняє окремі
   речовини в комбінованому препараті.

3. **Комбіновані препарати**: розкладаються на окремі речовини в
   Medications. Drug Interactions ніколи не бачить назву бренду —
   тільки список окремих речовин.

4. **Симетрія**: A+B — те саме, що B+A. Зберігаємо один запис,
   сортуємо дві речовини перед пошуком (не два дзеркальні записи).

5. **Роль системи**: показує факт відомої взаємодії (речовина,
   тяжкість, опис за джерелом). Не оцінює доречність застосування,
   не враховує контекст (вік, ургентність, off-label намір) —
   рішення лишається за лікарем.

6. **Джерело даних v1**: список з наукової статті Phansalkar et al.,
   2013, BMC Medical Informatics and Decision Making — відкрита
   ліцензія Creative Commons Attribution (CC BY), вимагає лише
   посилання на авторів. RxNav Interaction API (NLM) вимкнений з
   січня 2024 — не використовувати. DrugBank вимагає окремої
   ліцензії і зараз призупинив видачу academic-даних.

7. **Interaction Evidence View** — відповідь пацієнту складається
   з трьох окремих блоків, кожен зі своїм рівнем довіри:
   - verified_interaction (з офіційного джерела)
   - patient_note (особиста нотатка, ліміт 2000 символів,
     явно підписана "не перевірено системою")
   - prescription_history (факт спільного призначення з наявних
     даних Medications — дати прийому, без імені лікаря/лікарні;
     явне попередження: це НЕ доказ безпеки, НЕ заміна перевірки
     лікарем)

8. **Дані лікаря/лікарні**: не додаємо у v1. prescription_history
   будується лише на датах прийому з Medications.

9. **Джерело як окремий тип поля**: `knowledge_source_id`
   (наприклад `phansalkar_2013`), не вільний текст — легше додати
   друге джерело в майбутньому без міграції структури.

10. **Оновлення бази**: не зараз. Завантажуємо список один раз,
    як зробили з Reference Range Resolver. Повертаємось до питання
    оновлення, коли виникне реальна потреба.
