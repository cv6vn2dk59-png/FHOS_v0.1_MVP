# Commit Checklist

Перевір перед КОЖНИМ git commit, незалежно від розміру зміни.

## 1. Перевір git status на несподіванки
- Чи всі файли в списку очікувані для цієї конкретної зміни?
- Чи немає файлів з інших модулів, яких ти не редагував навмисно?
- Якщо бачиш `modified` файл, якого не чіпав — СТОП. Спочатку `git diff <файл>`, з'ясуй причину (Ситуація 1/2/3 нижче), лише потім продовжуй.

## 2. Три ситуації для неочікуваної зміни
1. **Очікувана** — забув, що зачепив цей файл → додай до коміту або зроби окремий логічний коміт.
2. **Стара незакомічена** → вирішуй окремо: залишити / закомітити окремо / відкотити. Не змішуй з поточною задачею.
3. **Випадкова** (typo в шляху, heredoc записав не туди) → `git restore <файл>` без вагань.

## 3. Перед комітом коду (не документації)Усі тести повинні пройти. Не комітити з падаючими тестами навіть "тимчасово".

## 4. Якщо змінена схема БД
Перевір, що head відповідає очікуваному ланцюгу ПЕРЕД `alembic upgrade head`, не після.

## 5. Прибери робочі артефакти
Одноразові перевірочні скрипти (`check_*.py` тощо) видаляй одразу після використання — не залишай у робочій директорії до моменту коміту.

## 6. Коміт — одна логічна зміна
Один коміт = одне завершене логічне рішення. Не змішуй непов'язані зміни (наприклад, фікс бага в Profile і нову фічу в Laboratory) в одному коміті.
